assignment-2

ASCII(아스키코드) UNICODE(유니코드)란 무엇일까?

컴퓨터는 정보를 표현하기 위한 기본 원리는 전기신호이다. 전기가 통하고 안통하는 방식으로 ‘1’ 과 ‘0’ 두가지 경우로 정보를 표현한다. ‘0’과 ‘1’을 데이터의 최소단위라고 하고 비트(bit)라고 부른다. 그런데 이 하나의 비트는 0과 1만 표현 할 수 있기 때문에 우리가 사용하는 문자를 표현하기엔 크기가 너무 작다. 그래서 ‘바이트’라는 단위를 사용하게 된다.

바이트란, 일정한 개수의 비트로 이루어진 연속된 비트열이다. 여기서 ‘일정한 개수’가 항상 정해진 건 아니지만, 최근에는 1바이트를 8비트로 간주하는 경우가 흔하다. 한개의 비트는 ‘0’과 ‘1’ 2개의 정보를 표현 할 수 있는데 이것을 8개로 묶으면 2^8개 즉, 256가지의 정보를 표현할 수 있다. 따라서 숫자와 영문자를 모두 표현할 수 있고 남는 공간에 특수 문자까지 할당할 수 있게 되었다.

ASCII(아스키코드)

그러나 컴퓨터는 우리가 말하는 a, b, c를 알지 못한다. 그래서 a는 1이고 b는 2다 라는 것을 미리 기기에 입력을 해두는 것이다. 그런데 어떤 기기는 a를 2로 표현하고 어떤 기기는 a를 1이라고 규정하면 글자가 서로다른 기기에서 다르게 출력될 가능성이 발생된다. 이러한 현상을 방지하고자 숫자와 문자를 매칭시키는 국제적인 규칙을 만들게 되는에 이 규칙을 아스키 코드라고 한다.

UNICODE(유니코드)

하지만 아스키 코드(ASCII)에는 영어만 존재한다. 여러나라에서 컴퓨터를 사용하게 되면서 다양한 표준들이 생기게 되며 이 많은 표준들을 하나로 합쳐야할 새로운 기준이 필요해졌다. 이러한 한계를 극복하고자 나온 것이 바로 국제표준 코드가 유니코드(UNICODE)이다. 1바이트로 모든 언어를 표현하기에는 부족함을 느꼈고 2바이트(2^16 =. 65,536)를 사용하게 되면서 많은 문자를 추가로 사용할 수 있게 되었다.

유니코드의 한계

하지만 영어와, 한글, 다른 특수문자를 표현할 때 각기 다른 바이트를 쓰게 되면서 컴퓨터에 큰혼란을 주었고 이러한 문제점을 극복하기 위해 다양한 인코딩 방식이 등장하게 되었다. 대표적인 변환 인코딩 형식에는 UTF-7, UTF-8, UTF-16, UTF-32 인코딩 등이 있다.


1byte는 8bit 일까?

ASCII(아스키 코드)의 표를 참고하면 이진수는 총 7자리이다. 나머지 한자리는 패리티 비트(Parity bit)라고 부르는데, 데이터의 에러를 탐지하기 위해 사용한다고 한다. 일곱자리의 이진수에서 ‘1’이 홀수개라면 끝에 1을 붙이고 ‘1’이 짝수개라면 끝에 0을 덧붙인다. 이렇게 패리티 비트를 정하여 데이터를 보내면 받는 쪽에서 수신된 데이터의 전체 비트를 계산하여 패리티 비트를 다시 계산함으로써 데이터 오류 발생 여부를 알 수 있다.

또한, 다양한 언어를 포괄하기 위해 비트를 확장한 이유도 있다. 아스키 코드표를 만들어 질땐, 영어권 사람들이 대부분 이용하였으며 이는 분명 모든 언어를 담기에는 한계가 발생되었기 때문에 이 두가지 이유로 인하여 8bit로 규정된 것이 아닐까 싶다.

  1. Pixel의 단위는 절대적 인가? 상대적인가?

먼저, 픽셀의 정의를 찾아보면 픽셀은 화면을 구성하는 가장 기본이 되는 단위이다. 우리가 모니터를 통해서 보는 문자와 이미지는 실제로는 픽셀이라고 하는 매우 작은 사각형의 점들로 구성되어 있습니다. 하나하나의 픽셀은 해당 위치에서 그 픽셀이 가진 색상 정보를 표현하게 되고, 그 픽셀들이 수없이 많이 모여서 우리에게 실물과 비슷한 수준의 이미지를 보여 줍니다.

해상도 ∝픽셀의 갯수, 픽셀의 밀도

1) 픽셀의 갯수

모니터의 경우 1024 x 768, 1900 x 1200 과 같은 각각의최적화된 해상도를 설정하여 사용하는데 이때 모니터에서 사용하는 해상도라는 용어도 가로 x 세로 픽셀의 갯수를 표현하는 용어이다. 이처럼 해상도라는 말은 어떤 이미지나 디스플레이를 구성하는 최소 단위인 픽셀의 총량을 나타내는 말입니다.

2) 픽셀의 밀도

픽셀의 밀도(집적도)를 나타내는 경우에도 해상도라는 용어를 사용합니다. 흔히 우리가 사용하는 모니터의 해상도는 72ppi라고 알려져 있는데, 1ppi란 Pixel per Inch의 약어로 1인치당 몇개의 픽셀이 들어가는지를 말합니다. 즉 1 x 1인치의 면적에 72 x 72개의 픽셀로 이루어져 있다는 뜻입니다.

(*dpi (Dots per Inch)라는 인쇄 용어는 1x1인치의 면적에 몇개의 점을 찍어 인쇄할 것인가를 나타내는 용어입니다.) 이처럼 해상도란 ppi와 dpi와 같이 픽셀의 밀도를 나타내는 용어입니다.

=> 해상도란 기본적으로 이미지가 포함하고 있는 정보와 같다고 볼 수 있다. 해상도가 높을 수록 더많은 이미지 정보를 갖고 있다는 것입니다. 이미지에 픽셀이 많으면 많을 수록 더 큰 이미지를 만들 수 있고, 또한 픽셀이 많으면 많을수록 픽셀의 밀도를 높여 더 좋은 품질의 이미지를 만들 수 있다는 것입니다. 결국 픽셀의 수는 많을 수록 좋은 것인데, 이 픽셀들을 얼마만큼의 밀도로 어떤 사이즈의 이미지를 표현할 것인가 하는 이슈가 바로 해상도 인것이다.

그렇다면 픽셀의 크기는 절대적인가 상대적인가?

픽셀의 이미지의 크기, 픽셀의 밀도, 픽셀의 총량에 따라 상대적이다!

위의 보이는 사진처럼 72ppi 이미지를 살펴보면, 1인치 면적 안에 518,400개의 픽셀로 이뤄져 있는 것을 알 수 있다. 만약 72ppi를 인쇄용도인 300ppi로 만들면서 픽셀수를 그대로 유지하기 위해서는 이미지의 사이즈가 2.4 x 2.4인치로 줄어드는 것을 확인할 수 있다. 즉 픽셀의 크기는 상대적이며 가변적인것을 확인 할 수 있다.

  1. 프로그램 언어를 Runtime 이전에 기계어로 해석하는 작업방식이다. (원시코드 => 목적코드(Object code)
  1. 런타임 이전에 Assembly 언어로 변환하기 때문에 구동시간이 오래 걸리지만, 구동된 이후는 하나의 패키지로 매우 빠르게 작동하게 된다. 구동시에 코드와 함께 시스템으로부터 메모리를 할당받으며 할당받은 메모리를 사용하게된다.
  1. 런타임 이전에 이미 해석을 마치고 대게 컴파일 결과물이 바로 기계어로 전환되기 때문에 OS 및 빌드 환경에 종속적이다. 그러므로 OS 환경에 맞게 호환되는 라이브러리와 환경을 구분해서 구축해줘야 한다.
  1. Compile 언어의 대표격으로 C / C++ 와 같은 언어들을 들 수 있으며, JAVA 역시 Byte code 로 바꾸기 위한 과정에서 컴파일을 수행한다.

2) 인터프릿 (interpret )

=> 소스코드의 각 행을 연속적으로 분석하며 실행

  1. 런타임 이전에 기계어로 프로그래밍 언어를 변환하는 컴파일 방식과 다르게, 런타임 이후에 Row 단위로 해석(Interpret)하며 프로그램을 구동시키는 방식이다.
  1. 프로그래밍 언어를 기계어로 바꾸지 않고 중간 단계를 거친 뒤, 런타임에 즉시 해석하기 때문에 바로 컴팩트한 패키지 형태로 Binary 파일을 뽑아 낼 수 있는 Compile 방식에 비해 낮은 퍼포먼스를 보이게 된다.
  1. 런타임에 직접 코드를 구동시키는 특징이 있기 때문에 실제 실행 시간은 느리며, 대신 런타임에 실시간 Debugging 및 코드 수정이 가능하다. 또한 메모리를 별도로 할당받아 수행되지 않으며, 필요할 때마다 할당하며 사용한다.
  1. 대표적인 Interpreter 언어로는 Javascript와 같은 스크립팅 언어들이 있다.
  1. 많은 프로그래밍 언어들의 인터프리터는 해석을 위한 Virtual Machine을 두고, Machine위에서 Interpret을 수행하게 되는데, 이때 해석의 기반이 되는 머신들이 OS 환경을 지원해줌으로써, 해당 방식으로 인터프리터는 OS및 플랫폼에 종속되지 않는 프로그램 구동이 가능하게 된다. (이런 특징을 지는 Interpreter는 Java의 JVM과 Python의 Analyzer가 있다.)

<컴파일러, 인터프리터 비교 chart>

  • 기계어로 번역된 목적 프로그램의 생성 여부에 따라 컴파일러 언어와 인터프리터 언어로 구분된다.
  • 인터프리터 언어는 번역 즉시 실행되기 때문에 해석 언어라고도 한다.

저급언어 어셈블리어 고급언어
기계가 알아듣기 쉬운 언어, 기계 중심의 저급언어 어셈블리어(Assembly)는 기계어와 1:1로 대응되는 기호로 이루어진 언어로, 니모닉(Mnemonic)언어라고도 함 사람 중심의 언어로 배우기 쉽다
2진수 0과 1만 사용하여 명령어와 데이터를 나타냄 하드웨어 제어에 주로 사용되며, 처리 속도가 빠름 호환성이 좋아 상이한 기계에서 별다른 수정 없이 실행가능한 언어다
컴퓨터가 직접 이해할 수 있어 실행속도가 빠름 기계어와 가장 유사하며 호환성이 떨어짐 실행을 위해서는 번역하는 과정이 필요하다
전문적인 지식이 없으면 이해하기 어려움 기계어로 번역하기 위해 어셈블리러(Assembler)라는 번역기가 필요함
기계마다 언어가 달라 호환성이 없음 어셈블리어에서의 상수의 정의 2진 상수 정의: BL DCBL3 “101” 16진 상수 정의 XL DC XL2 “A2”
